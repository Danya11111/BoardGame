# RUNBOOK: запуск и демонстрация BoardGame

## 1. Запуск «в один шаг»
1. Клонируйте репозиторий и откройте **PowerShell** в корне проекта (рядом с `pom.xml`).
2. Выполните **`.\run.ps1`**.

Дополнительно ничего настраивать не нужно, если у вас **установлен и запущен Docker Desktop** (или иной Docker с `docker compose` в PATH): скрипт сам скачает JAR lsFusion в `lib/`, при отсутствии создаст **`.env`** из **`.env.example`**, поднимет **PostgreSQL**, сервер логики и **веб-клиент**. Откройте **http://localhost:8080/lsfusion**. Параметры приветствия и UI в контейнере заданы в **`docker/logics/entrypoint.sh`** (`settings.enableUI=2`, `settings.enableAPI=1`).

**Принудительно без Docker** (локальный только сервер логики): **`.\run.ps1 -Local`**. Тогда нужны **Java 17** и **Maven** в PATH, а также доступный **PostgreSQL**. Скрипт сам создаст **`lsfusion.properties`** из примера и подставит пароль **`boardgame_dev`** вместо плейсхолдера; при наличии **`psql`** в PATH попытается выполнить **`scripts/postgres-init.sql`**, если базы **`boardgame`** ещё нет. Если пароль суперпользователя PostgreSQL у вас не `boardgame_dev`, один раз отредактируйте **`db.password`** в **`lsfusion.properties`**.

Опционально (и в Docker через `.env`, и локально в `lsfusion.properties`): **`boardgame.initialAdminEmail`**, **`boardgame.registrationUrl`** — см. **`README.md`**.

## 2. Проверка, что сервер поднялся
- **Docker:** в логах контейнера **logics** должны появиться строки вроде `Logics instance has successfully started` и `Server has successfully started`; веб отвечает по адресу выше.
- **Локально (`-Local`):** те же строки в консоли или в **`logs/start.log`**.

## 3. Сценарий гостя
1. Открыть веб-клиент: при **`settings.enableUI=2`** первым экраном должна быть форма **«Клуб настольных игр»** (`welcomeLanding` в навигаторе): заголовок, короткое описание, пример с Уно, подсказка про вход и главную, кнопки **«Войти»** и **«Зарегистрироваться»**.
2. **«Войти»** — стандартный диалог lsFusion (до этого логин не показывается как обязательный первый шаг). После успешного входа открывается **«Главная»** (`userHomeHub`) с двумя сценариями: «Найти похожую игру» и «Библиотека настольных игр».
3. **«Зарегистрироваться»** — при заданном `boardgame.registrationUrl` открывается системный браузер (десктоп) и показывается сообщение со ссылкой (удобно для веб-клиента); при отсутствии URL — понятное сообщение участнику и подсказка администратору про `boardgame.registrationUrl`.

## 4. Сценарий участника клуба (вошёл как обычный пользователь)
1. После входа по умолчанию доступна **«Главная»** (`userHomeHub`): два крупных действия с пояснениями под ними.
2. **«Найти похожую игру»** — форма с подсказками сверху; сузить список фильтрами, выбрать игру, затем **«Показать похожие игры»** или **«Открыть карточку игры»**.
3. **«Библиотека настольных игр»** — полный список с вводным текстом; фильтры, карточка и рекомендации как выше.
4. Убедиться, что в рекомендациях есть балл и пояснение (в т.ч. «Рекомендовано клубом» при ручной связи в демо-данных).

## 5. Сценарий администратора
1. Войти под учётной записью с ролью `admin`. Первого администратора можно получить, задав `boardgame.initialAdminEmail` и зарегистрировавшись или войдя с этим email (плюс смена email у существующей учётки). При каждом открытии «Главной» роль `admin` дополнительно синхронизируется с этим email (`syncClubHomeOnOpen` в `BoardGameAuth.lsf`, событие `ON INIT` у формы `userHomeHub` в `BoardGameUI.lsf`).
2. Папка **«Администрирование»** (структура может быть вложенной):
   - **«Каталог игр»** — список игр и кнопка **«Создать игру — форма с полями»** (сразу карточка с названием и остальными полями);
   - **«Связи для рекомендаций»** — только ручные связи «похожие» между уже существующими играми (не создание новой игры); слева список игр-источников, справа таблица кандидатов с **названием, жанром, диапазоном игроков** и галочкой «похожа на»;
   - **«Участники и роли»** — назначение и снятие роли администратора (на экране есть напоминания: нельзя снять с себя и с последнего администратора);
   - **«Справочники»** — жанры, издательства, владельцы, механики;
   - **«Демо и проверка»** — загрузка демо-данных;
   - **«Справочники: только просмотр»** — чтение без редактирования.
3. В карточке игры (из библиотеки или подбора) у администратора доступно **«Редактировать игру»**.
4. На **«Главной»** у администратора есть кнопки **«Добавить новую игру (поля карточки)»** и **«Каталог игр»**; у не-админов они скрыты. Папка **«Администрирование»** в навигаторе тоже только для админов.

## 6. Проверка демо-данных
1. При **первом** запуске с пустой БД демо подтягиваются автоматически (`onFinallyStarted` → `applyExampleData` без диалога `MESSAGE`, чтобы не блокировать сервер в Docker). Повторный запуск не дублирует записи (идемпотентно). При необходимости обновить вручную: **«Администрирование» → «Демо и проверка» → «Загрузить или обновить демо-данные»** — появится сообщение в UI.
2. Обязательные по ТЗ: **Мафия**, **Шпион**, **Бункер**. Дополнительно в каталоге клуба: **Уно**, **Свинтус**, **Лунаси**, **Экивоки**, **Пандемия**, **Колонизаторы**, **Каркассон**, **Манчкин**, **Имаджинариум**, **Алиас**, **Диксит**, **Кодовые имена**, **7 Чудес**, **Ticket to Ride**, **Azul**, **Монополия** (дубликата «Catan» нет — используется имя **Колонизаторы**).
3. Ручные связи для проверки пояснений: **Мафия → Экивоки**, **Уно → Свинтус / Лунаси**, **Диксит → Имаджинариум**, **Шпион → Кодовые имена**, **Каркассон → Колонизаторы** (в UI: «клуб отметил как похожую»).

## 7. Docker (детали)
- Обычный путь: **`.\run.ps1`** при запущенном Docker сам скачает **`lib/lsfusion-server-6.1.jar`** и выполнит **`docker compose up --build`**. Образ **logics** намеренно не качает JAR внутри Dockerfile.
- Ручной запуск без скрипта: перед **`docker compose build`** положите JAR в **`lib/`** (или один раз запустите **`.\run.ps1`**).
- **OCR (tess4j):** при `mvn compile` Maven копирует зависимости в **`target/server-lib`**; образ **logics** подключает их к classpath, в JRE-образе ставится пакет **`tesseract-ocr`**. Локально **`run.ps1`** тоже добавляет `target/server-lib/*.jar` к `-cp`.
- В `.env`: `BOARDGAME_INITIAL_ADMIN_EMAIL` → `-Dboardgame.initialAdminEmail=...`
- В `.env`: `BOARDGAME_REGISTRATION_URL` → `-Dboardgame.registrationUrl=...` (см. `docker/logics/entrypoint.sh`, `docker-compose.yml`).
- В `.env`: **`LSFUSION_LOGICS_HOST`** / **`LSFUSION_LOGICS_PORT`** — хост и порт RMI для контейнера **web** (по умолчанию `logics` и `7652` внутри compose). Если веб-клиент не в той же docker-сети, что logics, задайте, например, `host.docker.internal` или `127.0.0.1` по ситуации (см. `README.md`).

1. `mvn -q clean compile`
2. Кратко пройти сценарии разделов 3–5.

## 10. Частые проблемы
- Порт `rmi.port` уже занят — остановить второй `java.exe` или сменить порт в `lsfusion.properties`.
- Ошибка подключения к БД — проверить `lsfusion.properties` и запуск PostgreSQL.
- Кнопка «Зарегистрироваться» не открывает браузер — проверьте `boardgame.registrationUrl` и среду (на сервере без GUI `Desktop.browse` может быть недоступен); во веб-клиенте после действия показывается текст со ссылкой.
- **`UnknownHostException: logics`** у веб-клиента — Tomcat резолвит имя сервиса compose с хоста, где это имя не существует. Настройте `LSFUSION_LOGICS_HOST` для сервиса `web` или используйте `127.0.0.1` в дескрипторе контекста локального Tomcat (см. `README.md`).
- Сразу логин вместо приветствия — на сервере логики не задано `settings.enableUI=2` (локально: `lsfusion.properties`; Docker: уже в `docker/logics/entrypoint.sh`).
- После входа **пустой экран** — см. раздел в **`README.md`** про навигатор и **`FIRST`** у **`userHomeHub`** / **`welcomeLanding`**.
- **Сервер «завис» на старте после строки про автозагрузку демо** — в старых сборках внутри `onFinallyStarted` вызывался `MESSAGE` из загрузки демо; в актуальной версии автозагрузка вызывает только `applyExampleData()` (без модального сообщения), а кнопка в админке по-прежнему показывает сообщение пользователю.
